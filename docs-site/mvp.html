Objetivo do MVP

Entregar uma CLI Python funcional que:

01 Conecta a múltiplos MCP servers (local STDIO + remoto HTTP) configurados via orcha.json
02 Descobre ferramentas (tools/list) e as exibe formatadas no terminal
03 Executa ferramentas (tools/call) diretamente via CLI
04 Integra um LLM real (Anthropic Claude) que decide quais ferramentas usar
05 Expõe o protocolo JSON-RPC bruto em cada etapa com flag --verbose

O que está INCLUÍDO no MVP

Comandos CLI

ComandoDescrição
orcha list-toolsLista todas as tools de todos os servers habilitados
orcha describe <tool>Exibe o inputSchema completo de uma tool
orcha call <tool> --args '<json>'Executa uma tool diretamente, sem LLM
orcha chat "<pergunta>"Ciclo completo: LLM → tool_use → MCP → resultado
orcha chat --interactiveREPL de chat com histórico mantido entre turnos

Flags globais

FlagComportamento
--verbose / -vExibe JSON-RPC bruto formatado em cada step
--dry-runMostra o que seria feito sem executar
--config <path>Usa arquivo de config alternativo
--server <nome>Restringe a operação a um server específico

Configuração (orcha.json)

type Suporte a "local" (STDIO) e "remote" (HTTP)
enabled Campo por server — true / false
environment Variáveis de ambiente injetadas no processo do server
llm Configuração do provider LLM (provider, model)

Transports

STDIO

Servidores locais iniciados como subprocesso via stdin/stdout

HTTP (Streamable)

Servidores remotos via POST para o endpoint + SSE opcional

LLM

◆ Provider: Anthropic Claude via ANTHROPIC_API_KEY
◆ Modelos: qualquer Claude com suporte a tool_use
◆ Loop de orquestração com múltiplas rodadas até resposta final

Critérios de Aceite do MVP

O MVP está completo quando todos os cenários abaixo funcionarem:

Cenário 1 Descoberta de ferramentas 
orcha list-tools --verbose
JSON-RPC initialize (request + response) exibido para cada server
JSON-RPC tools/list (request + response) exibido para cada server
Lista formatada de todas as tools, agrupadas por server, exibida ao final
Cenário 2 Execução direta de tool 
orcha call filesystem_read_file --args '{"path": "./README.md"}' --verbose
JSON-RPC tools/call enviado ao server exibido
JSON-RPC de resposta do server exibido
Conteúdo do arquivo exibido como resultado
Cenário 3 Chat com orquestração LLM 
orcha chat "Quais arquivos existem neste diretório?" --verbose
Mensagem enviada ao LLM (com lista de tools) exibida
Resposta do LLM com tool_use (nome + argumentos) exibida
JSON-RPC tools/call enviado ao MCP server exibido
Resposta do MCP server exibida
Resposta final do LLM com base no resultado exibida
Cenário 4 Configuração mínima

Com o seguinte orcha.json:

{
  "llm": {
    "provider": "anthropic",
    "model": "claude-3-5-sonnet-20241022"
  },
  "mcp_servers": {
    "filesystem": {
      "type": "local",
      "command": ["npx", "-y", "@modelcontextprotocol/server-filesystem", "."],
      "enabled": true
    }
  }
}
Todos os comandos funcionam sem mais configuração além de ANTHROPIC_API_KEY
Cenário 5 Server desabilitado não aparece

Com "enabled": false em um server:

orcha list-tools
Server desabilitado não aparece na listagem
Nenhuma conexão é feita com aquele server

Definição de Pronto (DoD)

Uma história de usuário está pronta quando:

O comportamento descrito nos critérios de aceite funciona
A flag --verbose exibe as mensagens JSON-RPC relevantes para aquela operação
Erros são tratados com mensagens claras no terminal (sem stack trace exposto ao usuário)
O código tem tipagem estática (type hints Python)
Existe ao menos um teste para o caminho feliz
