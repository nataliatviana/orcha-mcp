Sprint 0 — Bootstrap

E0 Setup

Bootstrap do projeto: repositório, estrutura de pastas, tooling e configuração de qualidade de código.

US00

Setup do projeto

Alta MVP

Como desenvolvedor, quero um repositório Python com estrutura de pastas, dependências e tooling configurados, para começar a implementar funcionalidades sem overhead de setup.

  • pyproject.toml com uv, Python 3.11+, todas as dependências do MVP
  • Estrutura orcha/cli.py, orcha/config/, orcha/client/, orcha/llm/, orcha/display/
  • orcha --help disponível via uv run orcha --help após uv sync
  • ruff configurado para lint e format
  • mypy configurado com strict = true
  • Pre-commit hooks configurados (ruff, mypy)
  • .env.example com ANTHROPIC_API_KEY= documentado
Definição de Pronto
  • uv sync && uv run orcha --help funciona sem erros
  • uv run ruff check . passa sem warnings
  • uv run mypy orcha/ passa sem erros
  • uv run pytest executa sem falhas de configuração
Sprint 1 — Fundação

E2 Configuração

Leitura, validação e gerenciamento do arquivo orcha.json.

US01

Configuração de múltiplos servers via JSON

Alta MVP

Como desenvolvedor, quero definir múltiplos MCP servers em um arquivo orcha.json, para gerenciar quais servers usar sem alterar o código.

  • orcha.json lido da raiz do diretório atual por padrão
  • Suporte a type: "local" (STDIO) e type: "remote" (HTTP)
  • Server local: command (lista), enabled, environment (opcional)
  • Server remoto: url, enabled, headers (opcional)
  • Seção llm com provider e model
  • Erros de schema indicam o campo inválido (sem stack trace)
  • Ausência do arquivo exibe mensagem orientando o usuário
// orcha.json
{
  "llm": { "provider": "anthropic", "model": "claude-3-5-sonnet-20241022" },
  "mcp_servers": {
    "filesystem": { "type": "local", "command": [...], "enabled": true, "environment": {} }
  }
}
Definição de Pronto
  • Testes cobrem: config válida, campo ausente, tipo inválido, arquivo ausente
  • Tipagem estática completa em orcha/config/
  • Erros exibem mensagem legível (sem stack trace)
US02

Variáveis de ambiente por server novo

Alta MVP

Como desenvolvedor, quero definir variáveis de ambiente específicas por server no orcha.json, para passar credenciais e configurações sem expô-las globalmente.

  • Campo environment aceita objeto de strings
  • Variáveis são mescladas com o ambiente do processo pai (não substituem)
  • Variáveis em environment têm precedência sobre o processo pai
  • Nenhum valor de variável é exibido no terminal por padrão
  • Com --verbose, exibe apenas os nomes das vars injetadas (não os valores)
Definição de Pronto
  • Teste verifica que a variável está disponível no subprocesso
  • Teste verifica que o valor não aparece em nenhuma saída --verbose
  • Tipagem estática completa
US03

Ativar e desativar servers individualmente

Alta MVP

Como desenvolvedor, quero desabilitar um server no orcha.json sem removê-lo, para testar outros servers de forma isolada.

  • Server com "enabled": false não é inicializado em nenhum comando
  • list-tools não exibe tools de servers desabilitados
  • Flag --server <nome> restringe a operação ao server especificado
  • Erro claro se nome passado em --server não existir no config
Definição de Pronto
  • Testes cobrem: server desabilitado não aparece, --server filtra corretamente
  • Tipagem estática completa
US04

Path alternativo de config

Média MVP

Como desenvolvedor, quero passar um caminho alternativo para o arquivo de config, para usar diferentes configurações por projeto ou contexto.

  • Flag --config <path> aceita caminho absoluto ou relativo
  • Erro claro se o arquivo não existir no path especificado
  • Flag disponível em todos os subcomandos
Definição de Pronto
  • Testes cobrem: path válido, path inexistente
  • Tipagem estática completa

E1 Core MCP Client

Implementação do MCP Client com lifecycle completo: inicialização, capability negotiation e encerramento de conexão.

US05

Conexão STDIO com server local

Alta MVP

Como desenvolvedor, quero conectar o orcha-mcp a um MCP server local via STDIO, para ver o ciclo de inicialização JSON-RPC acontecendo.

  • Inicia o processo do server como subprocesso usando o command do orcha.json
  • Envia initialize com protocolVersion e clientInfo
  • Recebe resposta com capabilities e as armazena
  • Envia notifications/initialized após receber a resposta
  • Handshake completo sem erros com server-filesystem
Definição de Pronto
  • Teste de integração faz handshake com server real (ou server-everything)
  • Comando não encontrado exibe mensagem clara
  • Tipagem estática completa
US06

Encerramento gracioso de conexões novo

Alta MVP

Como desenvolvedor, quero que todas as conexões com MCP servers sejam encerradas corretamente ao finalizar, para evitar processos órfãos e respeitar o lifecycle MCP.

  • Ao finalizar qualquer comando, todos os subprocessos STDIO são encerrados
  • Encerramento aguarda finalização com timeout de 5s (depois força SIGKILL)
  • Ctrl+C (SIGINT) aciona encerramento gracioso antes de sair
  • Com --verbose, exibe mensagem de encerramento por server
Definição de Pronto
  • Nenhum processo zumbi após execução (verificável com ps)
  • Tipagem estática completa
US07

Exibição da capability negotiation

Alta MVP

Como desenvolvedor, quero ver no terminal as capabilities negociadas durante o handshake, para entender o que cada server suporta.

  • Exibe resumo formatado: nome do server, versão, capabilities declaradas
  • Com --verbose, exibe JSON-RPC raw completo do initialize
  • Sumário exibido antes de qualquer operação subsequente
Connected servers:
  filesystem v0.6.2 tools(listChanged=true)
  fetch v0.1.0 tools
Definição de Pronto
  • Teste verifica que nome, versão e capabilities aparecem no output
  • Tipagem estática completa
Sprint 2 — Discovery

E3 Tool Discovery

Descoberta e exibição das ferramentas disponíveis em todos os servers conectados.

US08

Listar todas as tools disponíveis

Alta MVP

Como desenvolvedor, quero executar orcha list-tools e ver todas as ferramentas disponíveis, para saber o que posso pedir ao LLM que ele use.

  • Conecta em todos os servers habilitados e chama tools/list
  • Tools agrupadas por server com namespace (server_toolname)
  • Exibe contagem total ao final
  • Com --server <nome>, lista apenas as tools do server especificado
Server: filesystem (5 tools)
  filesystem_read_file       Read the complete contents of a file
  filesystem_write_file      Create a new file or overwrite
  filesystem_list_directory  List the contents of a directory

Server: fetch (2 tools)
  fetch_fetch               Fetches a URL from the internet

Total: 7 tools disponíveis
Definição de Pronto
  • Teste de integração com server-filesystem valida output esperado
  • Tipagem estática completa
  • --verbose exibe JSON-RPC do tools/list (coberto por US11)
US09

Inspecionar o inputSchema de uma tool

Média MVP

Como desenvolvedor, quero ver o inputSchema completo de uma tool específica, para entender quais parâmetros ela aceita antes de chamá-la.

  • Comando orcha describe <tool_name> exibe o inputSchema formatado
  • Indica campos required e opcionais com destaque visual
  • Aceita nome com namespace ou com --server explícito
  • Erro claro se a tool não for encontrada
Definição de Pronto
  • Teste verifica output para uma tool conhecida
  • Tipagem estática completa

E6 Debug / Inspeção

Ferramentas transversais para inspeção do protocolo, ativas em todos os comandos via flags.

US10

Flag --verbose global

Alta MVP

Como desenvolvedor, quero que qualquer comando aceite --verbose / -v, para ver o JSON-RPC bruto em qualquer operação.

  • Funciona em list-tools, call, chat e describe
  • Requests em azul, responses em verde, notifications em amarelo
  • Prefixo de direção e server: → HOST→filesystem / ← filesystem→HOST
  • JSON pretty-printed com indentação de 2 espaços
  • Saída verbose vai para stderr, não para stdout
Definição de Pronto
  • Teste verifica que stdout não contém JSON-RPC
  • Tipagem estática completa
US11

Inspecionar JSON-RPC do tools/list e tools/call

Alta MVP

Como desenvolvedor, quero ver o JSON-RPC bruto trocado durante tools/list e tools/call, para entender exatamente o que o protocolo transfere em cada operação.

  • Com --verbose em list-tools: exibe request e response do tools/list por server
  • Com --verbose em call: exibe request tools/call com args e response completa
  • Texto longo é truncado em 500 chars com indicação de truncamento
  • JSON exibido é idêntico ao transmitido (sem modificações)
Definição de Pronto
  • Teste verifica que o JSON exibido é válido e contém campos esperados do protocolo
  • Tipagem estática completa
Sprint 3 — Execution

E4 Tool Execution

Execução direta de tools via CLI, sem passar pelo LLM.

US12

Executar uma tool diretamente

Alta MVP

Como desenvolvedor, quero executar orcha call <tool_name> --args '<json>', para testar uma tool individualmente sem um LLM decidindo.

  • Aceita nome com namespace (filesystem_read_file)
  • Argumentos como JSON string via --args '{...}'
  • Resultado formatado (texto, JSON ou tipo binário identificado)
  • Exit code 1 se a tool retornar isError: true
  • Erro legível se JSON em --args for inválido
Definição de Pronto
  • Teste de integração executa filesystem_read_file e valida conteúdo
  • --verbose exibe JSON-RPC da chamada (coberto por US11)
  • Tipagem estática completa
US13

Roteamento automático para o server correto

Alta MVP

Como desenvolvedor, quero que o orcha-mcp identifique automaticamente qual server possui a tool chamada, para não precisar especificar o server manualmente.

  • Roteamento pelo prefixo do namespace (filesystem_* → client filesystem)
  • Erro claro se tool não for encontrada em nenhum server
  • Erro claro se houver ambiguidade (mesmo nome em dois servers sem namespace)
  • Mapa de roteamento construído uma vez na inicialização
Definição de Pronto
  • Testes cobrem: roteamento correto, tool não encontrada, ambiguidade
  • Tipagem estática completa
US14

Modo dry-run

Média MVP

Como desenvolvedor, quero usar --dry-run para ver o que seria executado sem executar de fato, para validar meu entendimento antes de disparar operações reais.

  • Exibe o JSON-RPC que seria enviado ao server
  • Não estabelece nenhuma conexão real com servers MCP
  • Não consome créditos de API do LLM
  • Exibe aviso visível [DRY-RUN] no início do output
  • Disponível em call e chat
Definição de Pronto
  • Teste verifica ausência de conexões reais (mock de stdio_client)
  • Tipagem estática completa
Sprint 4 — LLM Loop

E5 Loop LLM

Integração com LLM real para orquestração completa: o LLM decide as tools, o orquestrador as executa, e o LLM responde com base nos resultados.

US15

Configuração do provider LLM

Alta MVP

Como desenvolvedor, quero configurar qual LLM usar no orcha.json, para testar com diferentes modelos sem alterar o código.

  • Suporte a provider: "anthropic" com ANTHROPIC_API_KEY
  • Campo model configurável (qualquer Claude com tool_use)
  • Erro claro e acionável se ANTHROPIC_API_KEY não estiver definida
  • Validação da API key na inicialização, não na primeira chamada
Definição de Pronto
  • Teste verifica mensagem de erro quando ANTHROPIC_API_KEY está ausente
  • Tipagem estática completa
US16

Chat com uma pergunta única

Alta MVP

Como desenvolvedor, quero executar orcha chat "<pergunta>", para ver o ciclo completo: LLM → tool_use → MCP → resultado → resposta final.

  • Conecta aos servers e descobre tools antes de chamar o LLM
  • Envia a pergunta com a lista completa de tools no formato tool_use
  • Detecta blocos tool_use e executa cada tool via MCP
  • Insere resultado como tool_result no histórico e repete o loop
  • Para quando o LLM retorna resposta sem tool_use
  • Trata caso em que o LLM responde diretamente sem acionar tools
  • Limite de segurança: máximo 10 iterações do loop
Definição de Pronto
  • Funciona com orcha chat "Quais arquivos existem neste diretório?"
  • Teste de integração valida ciclo completo (pode usar mock da API)
  • Erros da API Anthropic exibem mensagem legível (sem stack trace)
  • Tipagem estática completa
US17

Exibir cada etapa do loop LLM com --verbose

Alta MVP

Como desenvolvedor, quero ver cada etapa do loop de orquestração ao usar --verbose, para entender como o LLM e o MCP interagem.

  • Mensagem enviada ao LLM (histórico + contagem de tools)
  • Resposta do LLM com bloco tool_use (nome + args)
  • JSON-RPC tools/call enviado e resposta do server
  • Mensagem final ao LLM e resposta final
  • Cada passo numerado sequencialmente [STEP N]
[STEP 1] Sending to LLM (claude-3-5-sonnet-20241022)
  → 1 message(s) | 7 tools available
[STEP 2] LLM responded with tool_use
  → Tool: filesystem_read_file
  → Args: {"path": "./README.md"}
[STEP 3] Executing via MCP
  → HOST→filesystem: {"jsonrpc":"2.0","method":"tools/call",...}
[STEP 5] Final response
  O README.md contém informações sobre...
Definição de Pronto
  • Teste verifica que todos os steps são emitidos para stderr
  • Tipagem estática completa
US18 E1

Conexão HTTP com server remoto

Alta MVP

Como desenvolvedor, quero conectar o orcha-mcp a um MCP server remoto via HTTP, para comparar o comportamento entre transport STDIO e HTTP.

  • Faz POST para a URL do server com a mensagem initialize
  • Suporte a headers customizados via campo headers no orcha.json
  • Lida com respostas síncronas (200 JSON) e streaming (SSE)
  • Timeout configurável via campo timeout no config (padrão: 30s)
  • Erro claro se a URL não estiver acessível
Definição de Pronto
  • Teste verifica handshake HTTP com server de teste local
  • Erros de rede exibem mensagem orientada a ação
  • Tipagem estática completa
Sprint 5 — Polimento
US19 E5

Chat interativo (REPL)

Média MVP

Como desenvolvedor, quero executar orcha chat --interactive e ter uma sessão de chat contínua, para explorar o loop MCP em múltiplos turnos.

  • Loop no terminal com prompt >
  • Histórico completo mantido entre turnos da sessão
  • Comando /exit ou Ctrl+C encerra graciosamente
  • Comando /clear limpa o histórico mantendo a sessão
  • Exibe tools e servers ativos no início da sessão
  • Servers ficam conectados durante toda a sessão (não reconectam a cada turno)
Definição de Pronto
  • Ctrl+C não deixa processos órfãos
  • Tipagem estática completa
US20 E6

Sumário de sessão ao encerrar

Baixa MVP

Como desenvolvedor, quero ver um sumário ao fim de qualquer operação, para ter uma visão geral do que foi feito naquela sessão.

  • Exibe: servers conectados, tools descobertas, tools chamadas (nome + contagem), tempo total
  • Com --verbose, exibe contagem de mensagens JSON-RPC por server
  • Sumário vai para stderr (não interfere em pipes)

  Servers:   filesystem, fetch
  Tools:     7 discovered, 2 called
    filesystem_read_file  ×1
    filesystem_list_directory  ×1
  Duration:  3.2s
─────────────────────────────────────────
Definição de Pronto
  • Sumário aparece mesmo quando a sessão é encerrada por Ctrl+C
  • Tipagem estática completa
Pós-MVP

E7 Orquestração Multi-Agente pós-MVP

Um agente orquestrador que delega sub-tarefas a agentes especializados via MCP.

US21

Agente delegando para sub-agente via MCP

Alta (futuro)
  • Orquestrador inicia sub-processo orcha chat como MCP server
  • Sub-agente expõe tool run_task(instruction) via MCP
  • Ciclo completo observável com --verbose nos dois níveis

E8 Resources e Prompts pós-MVP

Suporte aos outros primitivos MCP além de Tools.

US22

Listar e ler Resources

Média (futuro)
  • Comando orcha list-resources: URI, nome, mimeType por server
  • Comando orcha read-resource <uri>: exibe conteúdo do resource
  • Com --verbose, exibe JSON-RPC de resources/list e resources/read
US23

Listar e usar Prompts

Média (futuro)
  • Comando orcha list-prompts: nome, descrição, argumentos por server
  • Comando orcha get-prompt <nome> --args '{...}': exibe prompt renderizado
  • Com --verbose, exibe JSON-RPC de prompts/list e prompts/get

E9 Comparativo entre Ferramentas pós-MVP

US24

Modo "side-by-side"

Baixa (futuro)
  • Flag --compare <tool> aceita opencode ou claude-code
  • Executa a mesma query nas duas ferramentas e exibe traces JSON-RPC lado a lado
  • Relatório de diferenças: round-trips, tools chamadas, latência
